## 1. 프로젝트 소개

### 개요

본 프로젝트는 터널 내 2차사고 예방을 목표로 하며, 이를 위한 수단으로 레일로봇을 제작하였습니다. 터널 내 2차사고는 일반 교통사고에 비하여 사망률이 굉장히 높습니다. 이에 저희 팀은 기존 2차 사고 대응 방식의 한계점을 분석하고, 이를 해결하는 것을 목표로 하였습니다.

최종적으로 산업통상자원부 장관상을 수상하였습니다.

### 기존 2차 사고 대응의 한계점

먼저 2차 사고에 대해 정의해보겠습니다. 2차 사고란, 선행 발생한 1차 사고 이후, 뒤따르던 후행차량이 앞에서 발생한 사고를 인지하지 못하여 발생하는 사고입니다.

그렇다면 현행 2차 사고 대응 방식은 어떠할까요? 기존 방식은 터널 입구에 배치된 터널 진입 차단 시설이나, 전광판(VMS), CCTV 모니터링을 통하여 차량 흐름을 제어합니다. 

그렇다면 기존 2차 사고 대응 방식의 한계점에는 어떤 것들이 있을까요?

먼저 기존 방식은 고정된 위치에서 사고 정보를 송출합니다. 하지만 이러한 방식은 다소 한계점을 가집니다. "운전자 단기기억 특성을 고려한 차내 교통안전정보의 효용함수 추정"[1]에 따르면 도로 주행중 제공된 교통정보의 효용은 약 22초간 유지됩니다. 즉, 사고 발생 정보를 송출할 시에 사고 발생 지점으로부터 거리가 멀다면, 사고 예방 효과가 적어집니다. 즉 사고 발생 지점으로부터 적절한 거리에서 반복적으로 송출할 시에 사고 예방 효과가 두드러집니다. 하지만 기존 고정된 위치에서 사고 정보를 송출하는 방식은 적절한 거리에서 반복적으로 사고 발생 정보를 송출하지 못한다고 판단하였습니다.

다음으로 기존 CCTV로 사고를 모니터링하는 방식은, 관리 인력이 상시로 모니터링 해야한다는 점에서, 감시 공백이 발생할 수 있다는 점에서 한계가 있습니다. 감시 공백이 발생한다면 사고 탐지 시간이 길어져 문제가 발생할 수 있다고 판단하였습니다. 더하여 설치할 CCTV 개수가 많다면 설치 비용, 관리 인력 등에 대한 비용적 문제 또한 발생할 수 있다는 점에 주목하였습니다. 이러한 기존방식의 한계점을 극복하여 경제적이고, 감시 공백을 최소화하는 것으로부터 사고 발생 시 탐지 시간을 줄여, 터널 내 2차사고 예방에 기여하고자 하였습니다.

이에 저희 팀은 이러한 한계점을 극복하기 위하여 레일로봇이라는 해결책을 선택하였습니다. 레일로봇은 구동부, 사이렌과 유사한 알람장치, 카메라, 서버 등이 주요 구성 요소입니다. 구동부는 기계적 안전계수 등을 면밀히 고려하여 완성도를 높였습니다. 더하여 직관적인 사고 발생 알람을 위하여 터널내 VMS(도로전광표지)와 연계되는 사이렌 장치를 도입하였습니다. 더하여 AI를 통한 모니터링을 구현하기 위하여 카메라를 설치하였습니다. 더하여 중앙 통제소에서 레일로봇을 관리하기 위하여 레일로봇과 통신하는 서버와 중앙통제소의 프론트엔드 페이지를 구현합니다.

### 레일 로봇 동작 시퀀스

로봇 동작을 설명하기 위하여 사고 발생 전, 사고 탐지, 사고 종료로 나누어보겠습니다. 

첫째로, 사고 발생 전입니다. 레일로봇등은 일정한 간격을 유지하며 사고 발생을 감지합니다. 또한 중앙 통제소에서 특정 부분을 탐지하는 것을 가능케하고자 레일로봇에게 특정위치로 이동할 수 있도록 합니다. 더하여 배터리가 부족하다면 레일 끝에 설치된 충전소에서 충전을 실시합니다.

둘째로 사고 탐지 입니다. 만약 1차 사고가 발생하여 레일로봇이 이러한 사고를 AI로 탐지한다면, 중앙 통제소에 경보를 보냅니다. 중앙 통제소는 위양성을 검사하여 실제 사고가 발생하였는지 검증합니다. 이때 실제 사고라고 판단하여 중앙 통제소가 로봇에게 신호를 전달한다면, 로봇은 사고 발생 지점으로부터 적절히 떨어진 지점에서 후행차량에게 전방에 사고가 발생하였음을 알립니다. 이후 관리자, 경찰, 구급대원을 통하여 사고를 처리합니다.

마지막으로, 1차 사고의 처리가 완료되면, 중앙 통제소는 사고 상황을 해제하고, 레일로봇이 평상시에 동작하는 상태로 유도합니다.

### 레일 로봇의 효용

이러한 레일로봇은 어떤 효용을 지닐까요?

먼저 레일로봇이 구동부를 통하여 유동적으로 움직일 수 있다는 점에서, [1]의 논문을 고려한, 적절한 위치에서의 알람을 도입하여 2차 사고를 예방하는 데에 효과적이라고 판단하였습니다.

다음으로 레일로봇 하나로 여러 CCTV를 대체할 수 있다는 점에서 효용이 높다고 판단하였습니다. 여러 CCTV를 대체하여 CCTV 설치 비용을 줄이고, 레일로봇이 대중화되어 가격이 낮아진다면, 충분히 경제적일 수 있다고 판단합니다. 

또한 AI를 활용하여, 감시공백을 최소화하여 인적 비용을 줄이고, 사고 탐지 시간을 줄일 수 있다는 점에서 효과적이라 판단합니다.

(물론 레일로봇이라는 방식보다는, 경제적이고 단순하며 공학적으로 아름다운~ 여러 예방 방법이 있을 것 입니다. 하지만 경진대회의 특성 상, 기술적 우수성을 보이는 것이 중요하다고 판단하여, 레일로봇을 해결 방안으로 선택하였습니다.)

    [1] 신소명,이수범,김형규,박민재,김경태,Shin So Myoung,Lee Soo Beom,Kim Hyung Kyu,Park Min Jai,and Kim Kyoung Tae. "동영상식 VMS로 사전정보제공시 터널 내 2차사고 감소효과에 관한 연구." 한국안전학회지 34.2 (2019): 56-62.

## 2. 라인트레이서 계수기 실패로부터 배운 교훈

이 프로젝트에서 저의 역할은 프론트엔드, 백엔드, 인프라 등을 관리하는 것이었습니다. 서버는 레일로봇이 중앙 통제소와 통신하며 지시를 받을 수 있도록 합니다. 프론트엔드 페이지는 로봇에서 서버로 전달된 정보(카메라, 현재 위치 등)을 관리자에게 직관적으로 전달합니다.

이전에 라인트레이서 계수기 서버를 운영하며 실패했던 경험이 있습니다. 문제의 원인을 분석하고 예방책을 학습하는 것, 이것만큼 중요한 것이 어디에 있을까요? 기존의 실패를 바탕으로 창의적 종합 설계 경진대회에서 서버를 운영하며 과거의 실수를 보완하여 서버를 운영하였습니다.

지금부터는 서버를 처음 운영하는 분들이 고려해봄직한 정보를 공유하려합니다.

### 캐싱

라인트레이서 계수기 서버에서는 서버 DB간 통신에서 부하로 인하여, 어플리케이션이 정상적으로 작동하지 못하였습니다. 즉, 부하를 줄이기 위하여 캐싱을 고려해야했습니다.
 
저희 팀은 로봇과 서버의 실시간 통신을 구현해야했습니다. 특정 임베디드 보드마다 적용 가능한 라이브러리가 상이할 수 있기에 범용적이고 구현하기 쉬운 Polling 방식을 택하였습니다. Polling 방식은 단순히 말하여, 주기적으로 api를 요청하되, 요청하는 주기를 짧게 줄여 실시간에 가깝게 정보를 공유하는 것입니다.

이때 여러대의 로봇이 서버에 동시에 접근한다면 부하가 많이 걸릴수도 있습니다. 이때 Polling의 경우 로봇이 서버에 데이터를 요청하면, 서버는 DB에 접근합니다. Polling 환경에서는 다수의 로봇이 짧은 주기로 API를 호출하게 되므로, 매 요청마다 DB에 접근할 경우 병목이 발생할 수 있습니다. 이때 부하가 많이 걸리는 지점은 보통 서버가 DB에 정보를 요청할때 발생합니다. 

그렇다면 DB에 접근하는 요청 자체를 줄이기 위해서는 어떤 방식을 택해야할까요? Polling, Listening, Long Polling이 있겠지만 시간적 여유가 없었기에, Polling으로 구현하며, 적절히 캐싱하는 방식을 택하였습니다.

로봇이 이전 주기에 데이터를 요청하여 서버가 응답한 시간이 0초이며 새로운 요청이 1초에 들어왔고 가정합시다. 이때 서버는 0.5초의 캐시 TTL(Time To Live) 이전 주기 로봇의 요청과 지금의 로봇의 요청이 0.5초를 넘는다면, DB에 접근하여 새로운 정보를 전송합니다. 만약 0.5초를 넘지 않는다면 서버 메모리에 있는 정보를 제공합니다.

이렇게 캐싱을 한다면 서버 - DB 간 통신에서 부하를 효과적으로 관리할 수 있어 보입니다.

### 부하 테스트

부하테스트란 서버가 시간당 어느정도의 요청을 견딜 수 있는지 파악하는 검사입니다. 예를들어 특정 api에 초당 100번의 요청이 발생하였을때, 이를 서버가 정상적으로 처리할 수 있는지 테스트하는 것이죠. 이때 k6과 같은 도구를 활용하여 부하 테스트를 진행할 수 있습니다. 초당 100번의 요청이 온다면 어느정도 비율로 응답이 지연되는지, 정상 응답과 비정상 응답 등을 파악하는 검사가 가능합니다.

이렇게 실제 서비스에서 발생할 부하의 maximum을 계산한뒤, 부하테스트를 진행하여, 실제 서비스에서 어플리케이션이 잘 동작할 수 있는지 예측해볼 수 있습니다.

## 3. Backend와 관련한 이모저모

### 인프라와 관련한 고민

엔지니어링을 할때에 가장 중요한 점 중 하나는, 제한된 리소스 내에서 효율적으로, 필요에 의해 자원을 분배해야한다는 것입니다. 이때 리소스란 팀이 가진 시간과 능력, 비용 등이 포함될 것 입니다. 즉, 제한된 시간 내에서 최선의 효율을 내는 것이 중요했습니다. 

이때, DB를 MongoDB Atlas를 이용할 지, 개인 서버 내에 MongoDB를 로컬로 운영할지 고민이었습니다. 아무래도 개인 서버 내에 MongoDB를 운영하는 것이 속도나 네트워크 측면에서 효율적이라 판단하긴 했습니다. 하지만 시간이 너무 촉박하여 기존에 활용했던 MongoDB Atlas를 유료 플랜으로 이용하기로 하였습니다. 즉, 비용을 조금 더 내더라도 시간적인 여유를 얻고자 하였습니다.(사실 이번 프로젝트 기간과 동아리 회장일을 병행하며 일이 폭탄이 되어버린 케이스이긴 합니다. 팀원들아 고마워요. 하트여 하트 ㅋㅋㅋ) 이렇듯 엔지니어링에서는 제한된 리소스 내에서 트레이드오프를 해야할 필요성이 있는 것 같습니다. 의사결정에 있어 이러한 판단 경험이 누적되는 것은 좋은 엔지니어로 성장하는 발판이 되는 것 같습니다.

### 에러 핸들링

(이후부터는 repo, serv, router 계층에 대해 기본적인 이해가 필요합니다. 인터넷에서 찾아보시는 것을 추천드립니다.)

타입스크립트(자바스크립트가 뭐죠?)에서 에러 핸들링 시에 try-catch 문을 사용하기도 합니다. 라인트레이서 계수기 프로젝트에서는
service와 repository 레이어 각각에서 try-catch를 사용하였습니다. 하지만 여러 계층에서 에러를 처리하다보니, 에러 흐름을 추적하기 어려웠습니다. 즉 에러 관리를 정돈할 필요가 있었습니다. 이에 이번 프로젝트에서는 service/repository 레이어에서 에러를 상위로 throw하고, router 계층에서 이를 중앙화하여 처리하는 구조를 선택하였습니다. 쉽게 말해, 상위 계층(service, repository)에서 라우터에 에러를 던지고, 라우터는 이를 처리합니다.

### 로깅

로깅, 생각보다 고려할 부분이 많아 보입니다. 장기적인 관점에서 로그가 쌓인다면 디스크 낭비가 될 수 있겠죠. 더하여 장애 발생시 로그에 필요없는 정보나 필요하지만 로그에 남지 않는다면 장애 발생시 사후처리를 신속히 진행할 수 없을 것 입니다. 즉, 로그를 통하여 빠르게 서버의 정보를 파악할 수 있으면서도, 저장된 로그의 효율을 유지하는 것 또한 고려대상이 될 수 있습니다. 또한 유지보수의 관점에서 여러 레이어(router, service, repository)에서 로깅을 무분별하게 하는 것이 아닌 특정 레이어에서만 로깅 중앙화하거나, error class를 만들어서 repository에서 service로 에러를 내려주는 등의 경우로 관리하는 경우도 있겠습니다.

로그의 정보를 빠르게 파악할 수 있는 방법에는 무엇이 있을까요? 바로 에러 계층을 나누는 겁니다. 자바스크립트의 경우 처음 접한다면, console.log나 console.error를 활용할 것입니다. 하지만 console.warning등 console의 매서드를 적절히 이용한다면, 빠르게 에러를 분류하여 볼 수 있겠죠. 더하여 로그가 저장되는 파일을 계층별로 나눈다면 더할나위 없을 겁니다.(+ AI로 로그를 관제한다면 베스트겠죠.) 하지만 여기에는 전제가 있습니다. 에러 계층을 어떤 기준으로 분류할 것이며, 어떤 정보를 담을지 정해두고, 개발할 때 이를 잘 지켜서 개발하는 것이겠죠.

더하여 로그에는 어떤 정보를 저장하는 것이 타당할까요? 저는 아직 정답을 확신하지는 못하겠습니다. 기본적인 시간, 응답 코드, api 등은 담겨야겠죠. 에러 로그에는 호출 스택이나, 코드 위치등도 담기는 것이 좋겠구요. 그런데 이런 부분은 실무를 겪어보고 논하는 것이 맞는 것 같기에 남겨두겠습니다.

### gzip 등 압축 방식

HTTP 통신에는 gzip 등의 압축 방식이 사용되기도 합니다. 결국 속도의 핵심은 전송되는 정보의 양 자체를 줄이는 것이 압도적으로 중요한 요소 중 하나라고 봅니다. 이에 좋은 압축 알고리즘을 선택하는 것이 중요하겠지요. 이 부분도 Nginx 등 웹서버에서 압축할 것이냐, 혹은 WAS에서 압축할 것이냐 등의 판단이 필요할 때가 있긴한데요, 보통 Nginx에서 압축 알고리즘을 적용하는 것으로 파악하고 있습니다. 

이때 로봇 즉, 저희 팀의 경우에는 임베디드 기기와 통신하며 얻은 경험이 있어 소개하려합니다. 여타 기기와 통신할 때에는 gzip이 아닌 zstd 등 다른 압축 알고리즘으로 전송해도 크게 문제는 없습니다. 그런데 임베디드 기기의 경우 라이브러리에 따라 zstd와 같은 압축 알고리즘이 적용되지 않는 경우가 있습니다. 즉, gzip과 같은 호환성 높은 압축 알고리즘으로 응답하는 것이 중요합니다. 그런데 이러한 압축 알고리즘이 Nginx에서 적용되는지, WAS에서 적용되는지 모른다면 문제가 생길 수 있겠죠. 즉, 어느 단에서 어떤 데이터가 정제되는지 미리 알고 있는 것이 중요합니다. 더하여 임베디드 기기에서 라이브러리를 사용할 때에 어떤 압축알고리즘을 지원하는지 미리 확인해보는 것도 좋아보입니다.

정리하자면 압축 알고리즘 선택 시 성능뿐 아니라 클라이언트 환경의 호환성 또한 고려해야합니다. 임베디드 기기와 통신을 준비하신다면 압축 알고리즘을 고려하고 명세 작성하시는 것도 방법이겠습니다. (더 좋은 압축 알고리즘 쓰면 비용이 줄어들 수 있겠죠.)

## 4. Frontend와 관련된 이모저모

본 프로젝트에서 프론트엔드는 관리자가 레일로봇에 지시를 내릴 수 있도록하고, 레일로봇의 정보를 관리자가 실시간으로 확인하게하는 데에 그 목적이 있습니다. 프론트엔드 페이지를 설계하며 겪었던 경험을 소개하겠습니다.

### 컴포넌트의 모듈화

컴포넌트를 적절한 단위로 나누는 것은 유지보수의 관점이나 특히 css 적용에 있어 결정적인 것 같습니다. 컴포넌트 모듈화를 잘 마무리 해놓았다면 css 측면에서의 유지보수에 용이해집니다. 특히 글자 tag, 테두리 tag 등을 기능별로 분리하여 적용하는 것이 좋았던 것 같습니다. 더하여 컴포넌트를 적절한 단위로 나눈다면, 재사용 측면에서도 이점을 볼 수 있겠죠. 너무 당연한 소리여서 쓸 필요가 있는지 의문이긴 합니다만, 적어두겠습니다.

엔지니어링에서의 가장 중요한 점은 단위를 적절히 나누는 것일테죠. 그리고 분리해야할 여러 단위를 하나의 단위에 몰아두어, 유지보수를 어려워짐을 방지하는 것이 중요합니다. 뻔한 소리긴 하네요.

### 디자인 변경

컴포넌트를 모듈화하여 얻은 이점에는 디자인 변경 시에 작업 시간이 단축되었다는 점입니다. 

프로젝트를 진행하며 기존 디자인에서 grafana 스타일의 디자인으로 변경해야하는 경우가 생겼습니다. 이때, 컴포넌트를 적절히 모듈화해놓으니, 윤곽은 유지하되 색깔정도만 바꿔서 빠르게 디자인을 변경할 수 있었습니다. 물론 디자인 변경 시에 컴포넌트 전체를 바꾸는 것이 타당하겠지만, 시간이 없었기에 위의 방식으로 컴포넌트를 재사용하였습니다.

이게 정말 이점인지는 모르겠지만, 여튼 디자인 변경 시에 컴포넌트를 적절히 모듈화 시켰기에 시간을 아낄 수 있었습니다.

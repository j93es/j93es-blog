## 0. 들어가며

좋은 프론트엔드 페이지란 무엇일까요? 여러가지 기준이 있겠지만, 이번 포스팅에서는 성능을 기준으로 좋은 프론트엔드 페이지를 정의해보겠습니다.

## 1. 성능 지표

먼저 프론트엔드 성능을 측정하려면 어떠한 방식이 있을까요? 성능 기준을 만들고 점수를 매기면 될 것입니다. 즉, 프론트엔드 성능 측정을 위해서는 성능 지표를 살펴보아야합니다. 각 성능 지표에 관하여 간단히 살펴보겠습니다. 

### 로딩 성능 

지표사용자가 페이지를 처음 열었을 때 얼마나 빨리 콘텐츠를 화면에 띄우고 인식하는지 측정합니다.

- LCP (Largest Contentful Paint): 화면에서 가장 큰 요소(이미지, 비디오 등)가 렌더링되는 시간. (목표: 2.5초 이하)

- FCP (First Contentful Paint): 텍스트나 이미지 등 첫 번째 콘텐츠가 화면에 그려지는 시간. (목표: 1.8초 이하)

- TTFB (Time to First Byte): 서버에 요청을 보내고 첫 번째 바이트가 응답될 때까지의 시간.

### 상호작용성 

지표사용자가 페이지와 상호작용(클릭, 타이핑 등)할 때 웹페이지가 얼마나 빠르게 반응하는지 측정합니다.

- INP (Interaction to Next Paint): 사용자의 모든 상호작용에 대한 전체적인 응답 지연 시간. (목표: 200밀리초 이하)

- FID (First Input Delay): 사용자가 처음 상호작용할 때(버튼 클릭 등) 브라우저가 응답하기까지 걸리는 시간 (INP로 대체되는 추세).

- TBT (Total Blocking Time): 페이지가 응답하지 못하게 막는 총 시간 (주로 개발자 도구에서 측정).

### 시각적 안정성 

지표페이지가 로딩되는 동안 화면 요소가 갑자기 움직여 사용자를 방해하는 정도를 측정합니다.

- CLS (Cumulative Layout Shift): 로딩 중 예기치 않게 레이아웃이 이동하는 누적 점수. (목표: 0.1 이하)

이렇게 프론트엔드 성능 지표를 나열해보았습니다. 그렇다면 이는 어떻게 측정할까요? 대표적으로 사용되는 툴은 구글의 Lighthouse입니다. 크롬 개발자 도구에 내장되어 있으니 한번 사용해보시길 바랍니다.

## 2. dynamic import

초기에 페이지를 마운트시키는 js파일의 용량이 크다면 어떤 일이 발생할까요? 페이지 로딩 자체가 느려질 것입니다. 일례를 들어봅시다. 블로그에는 포스팅 목록을 확인하는 메인 페이지와 블로그 글을 볼 수 있는 문서 페이지가 있습니다. 

이때 포스팅 목록을 확인하는 페이지는 굉장히 작은 용량으로 작성되어 있습니다. 반면 문서 페이지는 마크다운을 랜더링해야하기 때문에, 많은 패키지로 인하여, 용량이 큽니다. 이때 메인페이지에 접근하는데에 있어, 문서 페이지의 js를 로딩해야할 필요가 있을까요? 아닐 것 입니다. 즉, 메인 페이지의 js와 문서 페이지의 js를 분리시켜, 메인 페이지 로딩을 빠르게 만들 수 있습니다.

즉, 용량이 큰 특정 컴포넌트(혹은 용량이 큰 패키지를 사용하는 컴포넌트)를 다른 js파일로 빌드하여, 나누어 서빙하는 방식이 dynamic import 입니다.

## 3. 같은 기능을 한다면 적은 용량을 이용하자.

초기에 로딩되는 속도를 높이려면 어떤 방법이 효과적일까요? CDN을 아용하거나 캐싱을 적절히 활용하는 것 등 여러 방법이 있겠습니다. 하지만 이번 주제에서는 파일의 용량 자체를 줄이는 것에 집중해보겠습니다. 지금부터 리소스의 용량을 줄이는 방안에 대해 살펴보겠습니다.

### 프로젝트에 적용한 npm 패키지 검토

npm 패키지를 선택할 때에 있어 용량을 검토하는 것 또한 중요합니다. 예를 들어봅시다. 같은 기능을 하더라도 한 패키지는 많은 용량을 차지하는 반면, 다른 패키지는 많은 용량을 차지하지 않습니다. 그렇다면 당연 용량이 적은 패키지를 선택하는 것이 타당하겠죠. 이처럼 js 파일의 용량자체를 줄이기 위해서는 적절한 패키지를 선택하는 것이 중요하겠습니다. (한가지 첨언하자면, 패키지는 빌드된 시점을 기준으로 결정하는게 맞습니다. 빌드시에 패키지마다 쓰는 함수만 가져오는 경우도 있기에 빌드되고 js의 용량 자체를 확인하는 것이 필요합니다.)

더 자세한 내용은 [무분별한 npm 패키지 추가의 위험성](https://j93.es/posting/engineering/risks-of-indiscriminately-adding-npm-packages/risks-of-indiscriminately-adding-npm-packages.md)에 적어두었습니다.

### 로딩 바

로딩 바 이미지로 사용하는 경우 종종 있으실겁니다. 그런데 더 좋은 방식은 없을까요? 

로딩 바 이미지는 보통 gif를 사용합니다. 그런데 gif 용량이 전체 프로젝트의 자바스크립트와 맞먹는 경우도 종종 있을 겁니다. 즉, 프로젝트 전체 용량이 2배가 되는 것이죠. FCP 등의 지표가 하락함은 당연할 것입니다. 즉, gif로 로딩바를 구현하기보다는 HTML/CSS로 로딩바를 구현하는 것이 더욱 타당해보입니다.

### 이미지

웹페이지에는 다양한 이미지가 존재하곤 합니다. 그런데 용량의 측면에서, 용량이 적은 이미지 포맷을 사용하거나, 이미지를 다운그레이드하여 사용하는 것이 중요하다고 판단합니다. 이미지의 용량이 적어진다면, FCP/LCP 등의 지표가 향상될 수 있습니다.

먼저 이미지 포맷을 바꾸어봅시다. 요새는 압축률이 좋은 jpg 등을 많이 사용하는 것으로 파악합니다. 하지만 여기에서 조금더 나아가면, 간단한 로고의 경우 jpg보다 svg가 유리할 수 있다고 판단합니다. 즉, 하나의 포맷을 여러 포맷으로 변환해보고 가장 적합한 포맷을 선택하는 것도 좋아보입니다. 더하여 디바이스의 사이즈에 따라서 이미지의 해상도를 변환하는 방법도 있겠지만, 사실 이것은 캐싱의 측면에서 적합하지 않을 수도 있겠습니다.

다음으로 이미지 다운그레이드가 있습니다. 간단히 말하자면, 이미지의 해상도를 낮추는 것입니다. 만약 높은 용량의 이미지를 서빙하고 있다면, 이를 다운그레이드하여, LCP 지표의 성능을 늘리는 것도 좋아보입니다.

### 에러 페이지, noscript 페이지 압축

프론트엔드 페이지 압축하여 서빙되고 있는지 확인하셨나요? gzip과 같은 압축 알고리즘이 아닌, 리소스(html, js, css 등) 자체를 압축하는(+ 난독화) 알고리즘 또한 존재합니다. React의 경우 빌드한다면 자동으로 압축된 파일이 서빙됩니다. 

그런데 에러 페이지, noscript 페이지가 여러 이유(개발과정에서의 편의 등)에 의하여 따로 관리되고 있는 경우 또한 있을 겁니다. 그런 경우 압축하여 보내고 있지 않을 가능성이 높습니다. 즉, 예외 케이스를 처리하는 페이지 또한 압축하여 서빙하는 것이 좋다고 판단되기는 합니다.

사실 이러한 예외 페이지를 압축하여 서빙하는 것이 꼭 필요하냐고 묻는 분들이 많으리라 생각합니다. 이런 의문에 저도 필수적이라고 대답할 수는 없을 겁니다. 하지만 좋은 엔지니어이며, 리소스가 충분하다면 이러한 디테일을 챙기는 것(특히 프론트엔드)이 아름다운 공학을 추구하는 것이라고 생각합니다.

## 4. 이미지 lazy loading

이미지 태그가 랜더링된다면, 이미지가 다운로드되기 시작합니다. 이때 여러 이미지 태그가 동시에 랜더링된다면, 이미지가 한번에 다운로드되어 병목이 발생합니다. 더하여 사용자가 아직 보지도 않은, 즉 뷰 포인트(client 디바이스에 표시되는 html의 범위, 사용자가 보고 있는 범위)에 있지 않은, 이미지 또한 다운로드하게 됩니다. 이는 이미지 다운로드에 병목현상을 불러일으켜 성능에 악영향을 미칩니다. 즉, 이미지를 다운로드 시간이 증가합니다. 이는 LCP 등의 지표에 악영향을 미칩니다.

그렇다면 사용자 디바이스 뷰 포인트에 있는 이미지를 그때 그때 다운로드하여 사용한다면 어떨까요? 이러한 방식을 lazy loading이라고 합니다. 

```html
<img loading="lazy" src="image.jpg" alt="..." />
```

이러한 lazy loading을 사용한다면 초기 네트워크 사용량을 줄이고, 필요하지 않은 리소스를 꼭 필요할 때에만 다운로드할 수 있습니다.

## 5. 이미지 height, width 지정

프론트엔드 화면이 표출되었다고 가정합시다. 그런데 큰 이미지가 로딩 중일때를 상상해봅시다. 그렇다면 일단 브라우저에서 기본적으로 제공되는 작은 기본 이미지가 표출됩니다.

```
{제목}
|---|
|   |
|---|
{footer}
```

이런식으로 화면이 표출될 것 입니다.

그런데 저 작은 이미지가 다운로드가 완료되었다고 가정합시다.

```
{제목}
|-----------|
|           |
|           |
|  (이미지)   |
|           |
|           |
|-----------|
{footer}
```

사용자는 갑자기 화면이 밀려난 것 처럼 느낄 것 입니다. 이러한 사용자 경험은 지양해야할 요소 중 하나입니다. 부드럽고 안정되게 웹사이트 경험을 느끼게 하기 위해서는 화면이 아래로 밀려나는 것을 방지해야할 것입니다. 이러한 성능 지표를 CLS라고 합니다. 

그렇다면 어떠한 방식으로 이미지로 인하여 화면이 아래로 밀려나는 것을 방지할 수 있을까요? 바로 이미지 태그를 랜더링할 시에 미리 가로, 세로 길이를 지정하는 것입니다.

```
{제목}
|-----------|
|           |
|           |
|  (로딩 중)  |
|           |
|           |
|-----------|
{footer}
```

이렇게 가로 세로 길이를 지정해둔다면, 이미지가 차지할 자리를 이미지 태그가 미리 선점해두기 때문에 이미지 로딩 이후 화면이 아래로 밀려날 일이 없게 됩니다. 즉, CLS 지표에 긍정적인 영향을 미치게 됩니다.

## 6. footer 마운트 시점 조정

footer 마운트 시점을 조정하는 것 또한 특정 상황에서 유용할 수 있습니다. 본 블로그의 경우, 앞서 말한 바와 같이, 기본 레이아웃과 마크다운(md)을 html로 파싱하는 js를 분리하여 로딩하는 상황이었습니다. 따라서 포스팅에 접근하면 다음과 같이 렌더링 되는 상황이었습니다.

```
{제목 - 기본 레이아웃}
{로딩 바 - md->html js 로딩 중}
{footer - 기본 레이아웃}
```

이때 로딩 바가 사라진 후 마크다운이 로딩된다면, footer가 화면 하단으로 밀려나는 것으로 Lighthouse에서는 판단하더군요. 따라서 본문이 로딩되기 전까지, footer를 마운트하지 않는 것으로 CLS 지표를 개선한 바 있습니다. (사실 이러한 해결책을 제시할 때는 "왜 이 방식을 선택하였는가?"가 중요하긴 한데요... 이 경우는 이 방식 밖에 생각나지 않았습니다. 이러한 해결책이 공학적으로 아름다운지는 모르겠습니다.)

## 7. preload, preconnect

특정 리소스로 인하여 랜더링이 늦어진다면, 이를 어떻게해야할까요? 예를 듦을 통하여, 조금 풀어서 설명해봅시다. 특정 리소스가 다운로드 되는 시간이 지연된다고 가정합시다. 그런데 이 리소스가 웹페이지를 마운트/랜더링하는데에 꼭 필요한 요소입니다. 그런데 이 리소스의 로드가 지연됨으로 인하여 웹페이지 전체의 마운트/랜더링이 block됩니다. 그렇다면 이러한 현상을 어떻게 완화해야할까요?

먼저 preconnect를 이용하는 것입니다. preconnect는 다음과 같이 사용합니다.

```html
<link rel="preconnect" href="https://example.com" /> 

preconnect는 특정 도메인에서 리소스를 가져올 것이라는 것을 브라우저에게 미리 알리는 것을 의미합니다. 그렇다면 브라우저는 해당 도메인과 사전에 핸드셰이크를 일정 혹은 전체 부분 진행하여 해당 도메인에 접근하는 속도를 높입니다. 즉, 리소스를 가져오는 속도를 조금이나마 빠르게 만들 수 있습니다.

다음으로 preload를 이용하는 방식도 있습니다. preload는 다음과 같이 사용합니다.

```html
<link rel="stylesheet" href="styles/main.css" />
```

preload는 웹페이지에서 필요로 하는 리소스를, 랜더링 수명 초기에 가져오도록 합니다. 즉, 랜더링을 block 시킬 수 있는, 다시말해 랜더링에 필요한 리소스를 미리 로드하여, 랜더링 속도를 향상시킬 수 있습니다.

## 8. 나오며

지금까지 블로그를 개발하며 고민했던 성능 개선 방안에 대하여 소개했습니다. 특히 로드되는 리소스의 용량과 병목을 줄이는 방향의 방안을 많이 소개하였습니다.

사실 너무도 당연한 방안과 사실들의 나열일 것입니다. 그리고 이 포스팅과 같이 성능 향상 방안을 결과로만 나열했을때, 결과를 그저 따라하는 경우가 많으리라 생각합니다. 하지만 저는 결과를 따라하는 것이 과연 좋은 엔지니어인지에 대한 고민을 해보셨으면 좋겠습니다. 결국 좋은 엔지니어란 주어진 환경에서 여러 성능 향상 방안의 전체 집합의 요소를 찾고, 어떻게하면 적용할 수 있을지 고민하는 것 또한 하나의 중요한 자질이라고 생각합니다. 열심히 고민해보시길 바랍니다.

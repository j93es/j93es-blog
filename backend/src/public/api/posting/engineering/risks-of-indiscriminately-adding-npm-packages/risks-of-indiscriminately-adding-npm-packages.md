## 0. 들어가며

패키지 잘 사용하고 계신가요? 거의 모든 개발자들은 패키지를 추가하여 전체 프로젝트를 구현하고는 합니다. 예를 들어 `npm install {패키지 명}` 등을 통하여 말이죠. 그런데 패키지 추가를 남용하고 있지는 않을까요? 남용하고 있다면 어떤 문제점이 발생할까요? 혹은 같은 기능을 구현하는데 있어 여러 패키지 선택 옵션이 존재한다면, 어떤 기준으로 패키지를 선택해야할까요?

이번 포스팅에서는 라이선스와 성능(특히 프론트엔드)를 중심으로, 패키지의 적절한 추가와 패키지 선택에서의 고려할 점을 다룹니다.

## 1. 라이선스

패키지를 선택하는데에 있어 라이선스를 고려하는 것이 중요해보입니다. 이 맥락에서 라이선스란, 공개된 오픈소스 프로젝트의 사용 범위를 명시하는 것으로 볼 수 있습니다. 각 라이선스 별로 저작권, 특허권, 사용범위 등에 대한 내용이 상이합니다.

그렇다면 언제 라이선스를 고려하면 좋을까요? fun한 프로젝트, 즉 개인 프로젝트에서는 상관 없을겁니다.(사실 명확히 하자면 이 경우에도 라이선스 고려하셔야합니다. 특정 라이브러리를 사용하였을때, 전체 프로젝트를 공개해야하는 경우가 있겠네요.) 하지만 돈이 들어오면 이야기가 반전되겠죠. 라이선스 다들 무심코 넘기셨을테지만, 한번쯤은 짚고 넘어가야할 사항일 겁니다.

먼저 라이선스를 분류하며 대표적인 라이선스에 대해 알아봅시다.

첫째로 퍼미시브입니다. 가장 자유롭게 코드를 사용할 수 있는 분류입니다. 이에 해당하는 라이선스는 MIT, Apache, BSD 등이 있습니다.

둘째로 약한 카피레프트입니다. 이 분류는 오픈소스 라이브러리를 수정하여 사용한다면, 수정한 부분은 게시하고, 해당 라이브러리를 링크하여 사용하는 소프트웨어는 공개하지 않아도 됩니다. 이에 해당하는 라이선스는 LGPL, MPL 등이 있습니다.

셋째로 강한 카피레프트입니다. 이 분류는 해당 오픈소스 라이브러리 또는 이를 수정한 것을 링크하여 사용한다면, 전체 프로그램의 소스코드를 동일한 라이선스로 공개해야하는 경우입니다. 이에 해당하는 라이선스는 GPL, AGPL 등이 있습니다.

더하여 이러한 라이선스에는 오픈소스를 사용하는 자에게 법적 권한 및 의무 그리고 준수해야 할 사항이 명시되어 있습니다. 예를 들어 사용 및 수정, 상업적 이용, 저작권 표시, 공개 의무, 법적 책임 등이 있겠습니다.

## 2. 그래프를 그리는데 몇 kb를 태워야할까? - 창의공학설계경진대회

패키지의 용량, 생각보다 중요합니다. 같은 기능을 구현하는데 더 적은 용량의 패키지를 이용하는 것이 좋겠지요. 프로젝트를 진행하며 겪었던, 패키지 용량에 대한 경험을 소개하려합니다.

이전 프로젝트인 창의공학설계경진대회에서 프론트엔드 페이지에서 그래프를 표시해야 했습니다. 제가 이 프로젝트를 진행할 당시, 그래프 그리는 패키지는 많았습니다. 문제는 대부분의 그래프 라이브러리의 용량이 상당히 크다는 점이었습니다. 패키지의 용량이 커진다면, 프론트엔드 페이지 로딩 속도가 느려질 수 있습니다. 해당 프로젝트는 관리자 페이지를 제작하는 것이었기 때문에 프론트엔드 성능이 크게 중요하진 않았지만, 공부하는 입장에서 신경써서 좋지 않을 이유는 없죠. 

저는 airbnb에서 개발한 visx를 사용하였습니다. 아주 가볍고, 기본적인 그래프를 그리는데에 좋은 패키지로 판단하였습니다. 지금은 더욱 좋은 패키지가 많을 수 있겠네요.

결론적으로 같은 기능을 구현할때, 라이브러리의 선택지가 많다면, 선택의 기준에서 용량을 고려함이 좋아보입니다. 용량이 적은 라이브러리를 선택하는 것은 생각보다 성능 향상에 중요합니다. 성능 향상에는 여러 방법이 있겠지만, 데이터 이동량을 줄이는 것이 상당히 효율적이라고 판단합니다.

## 3. 맘에 안든다면 만들어 쓰자. - gray-matter

패키지 꼭 있는 것만 써야할까요? 필요 이상의 기능을 지원하거나, 용량이 너무 크거나, 라이선스가 용도에 맞지 않거나, 공학적으로 아름답지 않은 패키지라면 만들어쓰는 것이 좋지 않을까요? 블로그를 개발하며 겪었던 경험을 소개하려합니다.

블로그를 개발할 때에 아래처럼 작성된 마크다운 형식의 파일에서 메타데이터와 본문을 파싱해야하는 필요가 생겼습니다.

```markdown
---
title: 제목
date: 날짜
tag: [Frontend]
category: engineering
description: "설명"
---

본문
```

이때 gray-matter라는 라이브러리를 사용할 수 있었습니다. 해당 라이브러리를 통하여 메타데이터를 객체 형태로 파싱할 수 있었습니다.

하지만 여러 문제가 있었습니다. 첫째로, (제가 잘못 적용했는지는 모르겠지만) 특정 typescript 환경의 React에서 해당 라이브러리가 호환되지 않는 것으로 파악하였습니다.(아마 조금 더 찾아보아야할 것 같습니다.) 둘째로, gray-matter는 제가 필요로하는 기능보다 많은 기능을 지원하며, gray-matter 라이브러리의 여러 의존성이 따라오기에, 전체 프론트엔드 프로젝트의 용량이 커진다는 문제가 있었습니다. 즉, 필요없는 기능으로 인하여 용량이 늘어나게 되었습니다. 다시말해, gray-matter가 상당히 많은 용량을 차지하여 성능 하락의 우려가 커졌습니다. 

제가 필요한 기능은 YAML frontmatter를 단순 파싱하는 정도였기 때문에, 복잡한 markdown AST 처리나 플러그인 시스템은 필요하지 않았습니다. 이에 마크다운 파싱 툴을 직접 개발하기로 하였습니다. 파싱 방법이야 메타데이터 작성 형식이 정해진다면, 코드를 작성하는 데에 있어 크게 어렵지 않으리라 판단합니다. 이를 통하여 gray-matter를 사용하는 것 보다 많은 용량을 줄일 수 있었고, 호환성 측면에서도 잘 적용되었습니다. 이처럼 라이브러리의 용량이 크고 필요없는 기능을 지원한다면, 직접 만들어 쓰는 것이 상당히 좋은 방법이라고 생각합니다. 특히 회사라면 라이선스 측면에서도 안정적일 수 있죠.

## 4. 나오며

무분별하게 npm 패키지를 추가한다면 라이선스, 성능 측면에서 불이익을 얻을 수 있습니다. 돈을 다루는 회사에서 라이선스에 대한 고려는 필수적일 수 밖에 없죠. 더하여 필요없는 기능을 덕지덕지 붙인 라이브러리의 경우, 전체 프로젝트의 용량이 증가하게 됩니다. 이는 특히 프론트엔드 성능에는 큰 차이를 가져올 수 밖에 없습니다. 

물론 모든 기능을 직접 구현하는 것이 항상 좋은 선택은 아닙니다. 충분히 검증된 라이브러리를 사용하는 것은 개발 속도와 안정성 측면에서 큰 장점이 있습니다.

중요한 것은 패키지를 습관적으로 추가하는 것이 아니라, 프로젝트의 목적과 요구사항에 맞게 적절한 선택을 하는 것이라 생각합니다.

이 글을 읽는 여러분 또한 무분별하게 npm 패키지를 추가하지 마시고, 여러 측면을 고려하여 npm 패키지를 추가하시길 바라겠습니다.

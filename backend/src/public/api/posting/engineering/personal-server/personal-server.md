## 0. 들어가며

이번 포스팅에서는 네트워크 전반에 대해 간략하게 다룹니다. 특히 집에서 서버를 구동하고자하시는 분들에게 도움이 될 만한 정보를 드리려합니다.

처음 서버를 구동하고자 할 때에, 어떤 정보를 어디서부터 공부해야할지 고민이 많으리라 생각합니다. 제가 서버를 구동하며 얻은 경험과 정보를 토대로, 서버를 구동하고자 하시는 분들에게 유익한 정보를 드리려합니다.

## 1. 브라우저에서 example.com에 접속하면?

브라우저에서 example.com의 웹페이지에 접근할때 어떤 과정을 거칠까요? 

이때 선행적으로 고려해야하는 요소가 있습니다. 브라우저에서 example.com 웹페이지를 랜더링할때 어떤 정보가 필요한지 파악해야합니다. 이는 html, css 등등이겠죠. 즉, example.com을 운영하는 서버(특정 포트 ex 80, 443)에서 html, css 등등을 받아와야할 것입니다. 그렇다면 example.com을 운영하는 서버의 특정포트에 어떻게 접근해야할까요? 전체적인 흐름을 먼저 제시하겠습니다.

1. DNS로 IP를 찾는다
2. 인터넷 라우팅으로 서버까지 간다
3. 서버의 특정 포트(80/443)에 연결한다
4. 웹서버, WAS가 요청을 받는다
5. HTML/CSS를 응답한다

이제 각 단계를 차례로 자세히 살펴봅시다.

### IP 주소란?

먼저 example.com에 해당하는 IP주소를 받아와야합니다. 이는 DNS가 담당합니다. 그런데 IP 주소란 무엇일까요? '택배 상자'(패킷, 쉽게 말해 요청)를 보낼 때, '아파트 단지' 주소가 필요한 것처럼, IP 주소는 '아파트 단지' 주소라고도 볼 수 있습니다.

IP 주소에는 크게 내부 IP, 공인 IP가 있습니다. 은유적으로 표현해보겠습니다. 아파트 단지에는 여러 동과 호수가 있습니다. 공인 IP는 '아파트의 동'에 해당할 것 입니다. 내부 IP는 '아파트의 호수'에 해당할 것입니다. 사실 정확히 1:1 대응되는 개념은 아니기에 각각에 대해 살펴봅시다.

먼저 공인 IP에 대해 알아봅시다. 공인 IP는 전세계에서 사용되는 인터넷망 전체에서 중복되지 않는, 즉 고유한 IP입니다. 예를 들어 example.com 서버가 특정 공인 IP를 가지고 있다면, 우리는 해당 IP를 통하여 example.com 서버에 접근할 수 있는 것이지요.

다음으로 내부 IP란 무엇일까요? 예를 들어봅시다. 먼저 하나의 와이파이 공유기에 여러 기기가 접속하여 있다고 가정합시다. 이때 공유기(NAT)는 각각의 기기(mac 주소)에 IP를 분배합니다. 하지만, 전세계에서 고유한 공인 IP를 각각의 기기에 분베할 수는 없습니다. 이에 내부 IP 대역을 지정하여 공인 IP와 분리하여 사용합니다.

IPv4 주소의 형식은 xxx.xxx.xxx.xxx로 나타낼 수 있습니다. 이때 내부 IP는 다음 3가지 대역에 해당합니다.

- 가장 큰 사설 IP 대역
`10.0.0.0 ~ 10.255.255.255`

- 중간 규모 네트워크용 사설 IP 대역
`172.16.0.0 ~ 172.31.255.255`

- 가정용 공유기 기본값으로 많이들 사용하는 사설 IP 대역
`192.168.0.0 ~ 192.168.255.255`

즉, IP 대역을 나눔으로써 사설망과 외부망을 구분합니다.

이때 DNS를 통하여 받는 IP는 당연 외부 IP입니다. 이렇게 DNS를 통하여 IP를 받아왔다면, 이제 실제 서버에 접근해야합니다.

### 라우팅

먼저 요청을 하는 컴퓨터가 있다고 가정합시다. 컴퓨터는 공유기로 요청을 전송합니다. 공유기는 해당 IP가 내부의 IP인지 확인하고, 외부 IP라면 ISP 등에 요청을 넘깁니다. 이러한 라우팅을 거쳐서 말단(대부분 서버 혹은 네트워크 장비)에 요청이 도착합니다.

즉, 라우팅은 라우터가 목적지 IP를 보고 내부인지 혹은 외부인지 파악하며, 다음 전달 위치(next hop)를 결정하는 과정이라고 볼 수 있습니다.

### 포트란?

이렇게 우리는 특정 요청을 목적지 IP에 해당하는 서버에 전송하는 과정을 알았습니다. 그런데 html을 응답받으려면 서버 내의 웹서버, WAS가 동작하는 포트로 찾아가야합니다. 즉 포트에 대해 먼저 알아봅시다.

컴퓨터가 네트워크상에서 데이터를 주고받을 때, IP 주소가 목적지 '아파트의 호수'라면 포트는 해당 '집 내의 방'와 같습니다. 하나의 컴퓨터 안에서 실행 중인 여러 프로그램(웹 브라우저, 메일 등)을 구분하여 데이터를 정확히 전달하기 위한 번호 체계입니다.

HTTP의 경우 80번, HTTPS의 경우 443번 포트를 관습적으로 사용합니다.

보통 하나의 포트에 하나의 어플리케이션이 동작합니다.(Nginx, 웹서버, WAS 등) 즉, example.com의 html, css를 받아오기 위해서는 포트에서 운용되는 WAS나 웹서버에 접근해야합니다. 즉, 요청할 때, ip 뿐만 아니라 포트번호 또한 포함하여 요청합니다. (사실 패킷에는 굉장히 많은 정보가 포함되어있지만, 간단히만 다루어보려하니 이 정도까지만 다루겠습니다.)

이렇게 요청이 example.com 서버의 특정 포트에 전달된다면, 해당 서버는 오류 등 예외 상황이 없다면 응답을 보냅니다.

## 2. 집 서버는 어떻게 외부에 공개될까?

지금까지 하나의 요청이 특정 서버로 전달되어 HTML, CSS가 응답되는 과정을 살펴보았습니다. 그렇다면 직접 서버를 운영할 수는 없을까요? 나만의 웹페이지를 만들어서 외부에 공개할 수는 없을까요? 개인용 서버를 운영에 대한 이론을 학습해봅시다.

준비물은 돌아다니는 컴퓨터(리눅스면 더 편할겁니다.) 하나와 공인 IP(집에 인터넷 설치되어있다면 대부분의 ISP에서 외부IP를 배정해주긴할 것입니다.)입니다.

### 외부 IP 확인

먼저, 나만의 웹페이지를 공개하기 위해서는 공인 IP가 있어야만 합니다. 왜냐하면 고유한 IP가 있어야지만, 전세계에 있는 임의의 사람이 내 서버를 찾을 수 있도록 해야하기 때문이죠. 즉, 공인 IP를 찾아야만 합니다. 

먼저 공유기에 컴퓨터를 연결합니다. 그리고 해당 컴퓨터로 [ifconfig.me](https://ifconfig.me/)에 접속합니다.(다만 브라우저나 특정 설정 등으로 다른 결과나 외부IP가 아닌 다른 IP가 출력될 수 있으니 주의하시길 바랍니다.) 이 외에도 공인 IP 찾는 방법은 인터넷에 널려 있으니 확인해보시길 바랍니다.

그런데 집 내부의 공유기가 또다른 NAT 밑에 있다면, 공인 IP가 있더라도 서버를 열어둘 수는 없을 겁니다. 하지만 이런 경우는 큰 사업장이 아닌 이상 별로 없기 때문에 넘어가셔도 좋을 것 같습니다.

### 포트포워딩

이렇게 외부 IP를 찾았다면, 공유기와 서버의 특정 포트를 연결해야합니다. 즉, {공인 IP}:{80번 포트} -> {서버의 8080 포트}를 연결해야하는 것이죠.

먼저 서버를 공유기에 연결했다면, 공유기가 해당 서버에 **어떤 내부 IP를 부여했는지 확인**해야합니다. 그리고 **공유기의 특정 포트를 서버의 특정 포트와 연결**해야합니다.

이건 공유기 별로 확인/설정방법이 다르긴합니다. 그런데 공통적인 구조는 비슷하기에 설명해보도록 하겠습니다.

먼저 공유기가 서버에 어떤 내부 IP를 부여했는지 확인해봅시다. 공유기 관리자 페이지에 접근해봅시다. 내부 네트워크 설정 등과 같은 메뉴에 들어가시면, 공유기에 연결된 기기에 할당한 IP가 보일 것입니다.(대부분 192.168로 시작한다고 보면 됩니다.) 이렇게 나열된 기기 목록에서 서버의 IP를 확인하시면 됩니다. 

더 고급으로 들어가자면 특정 기기에 내부 IP를 고정시킬 수 있습니다. 예를 들어 서버에 랜선을 빼고 다시 꽂는다면, 서버에 할당된 내부 IP가 변경될 수 있습니다. 즉, 특정 기기에 내부 IP를 고정시키는 작업을 통하여 랜선을 빼고 다시 꽂아도, 기기의 IP가 변경되지 않도록 할 수 있습니다. DCHP 서버 설정이라는 메뉴에서 설정 가능합니다.

다음으로 공유기의 특정 포트를 서버의 특정 포트와 연결하는 포트포워딩 설정 방법을 확인해봅시다. 대부분의 공유기 제조사가 포트포워딩이라는 메뉴를 가지고 있을 겁니다. 해당 메뉴에 접속합시다. 여기에서 외부 포트와 내부 IP 및 포트를 연결하는 화면이 있을 겁니다. 여기에 외부에서 들어오는 포트를 지정하고, 이전에 확인한 서버의 내부 IP 및 어플리케이션을 구동할 포트를 적으면 됩니다.

이렇게 외부에서 들어오는 요청을 서버의 특정 포트로 보낼 수 있는 기반을 마련하였습니다.

### Nginx/방화벽 설정

그런데 서버의 포트에 HTML, CSS를 서빙하는 웹서버나 WAS가 있어야할 것으로 보입니다. 간단하게 웹서버를 구동시켜봅시다.

웹서버의 대표적인 종류에는 Nginx, Apache 등이 있습니다. 웹서버는 특정 디렉토리에 있는 `html`, `css`, `js` 파일을 읽어서 브라우저에 전달하는 역할을 합니다. 저는 개인적으로 Nginx를 즐겨쓰기에 Ubuntu + Nginx를 기준으로 설명해보겠습니다.(사실 정말 간단하게 하려면 node 서버만 올려도 무방할 겁니다.)

#### 1) Nginx 설치

```bash
sudo apt update
sudo apt install nginx
```

설치 후 실행 상태 확인:

```bash
systemctl status nginx
```

정상이라면:

```bash
active (running)
```

이 보입니다.

#### 2) 기본 웹 디렉토리 이해

nginx는 기본적으로 아래 디렉토리를 웹 루트로 사용합니다.

```bash
/var/www/html
```

즉:

```bash
/var/www/html/index.html
```

파일이 존재하면:

```text id="7pm74o"
http://서버IP
```

접속 시 해당 html을 보여줍니다.

더하여 html을 변경한다면 Nginx 서버를 리로드해야 변경된 html이 반영되는 경우도 있습니다.

```bash
sudo systemctl reload nginx
```

를 통하여 Nginx 서버를 리로드하시길 바라겠습니다.

#### 3) 테스트용 HTML 만들기

기존 파일 제거 후:

```bash
sudo rm /var/www/html/index.nginx-debian.html
```

새 html 생성:

```bash
sudo nano /var/www/html/index.html
```

예시:

```html
<!DOCTYPE html>
<html>
<head>
  <title>My Server</title>
</head>
<body>
  <h1>Hello Nginx</h1>
  <p>내 서버가 정상적으로 동작중입니다.</p>
</body>
</html>
```

저장 후 브라우저에서:

```text
http://서버IP
```

접속하면 페이지가 보입니다.

#### 4) Nginx 설정 파일 이해

핵심 설정 파일:

```bash
/etc/nginx/sites-available/default
```

열어보면 대략 이런 형태입니다.

```nginx
server {
    listen 80;

    server_name _;

    root /var/www/html;
    index index.html;

    location / {
        try_files $uri $uri/ =404;
    }
}
```

중요한 항목만 보면:

| 설정         | 의미                  |
| ---------- | ------------------- |
| listen 80  | 80번 포트에서 HTTP 요청 수신 |
| root       | html 파일 위치          |
| index      | 기본 파일               |
| location / | 요청 처리 방식            |

즉:

```text
브라우저 → nginx:80 → /var/www/html/index.html
```

흐름입니다.

#### 5) 설정 변경 후 적용

설정 수정 후:

```bash
sudo nginx -t
```

문법 검사.

정상이면:

```bash
sudo systemctl reload nginx
```

재시작 없이 설정만 반영됩니다.

#### 6) 방화벽 허용

이때 80번 포트로 들어오는 요청을 정상적으로 수신하려면 80번 포트로 들어오는 패킷을 차단하는 방회벽을 해제해야합니다. Ubuntu에서 방화벽 설정은 UFW, iptable 등을 이용합니다.

Ubuntu UFW 사용 시:

```bash
sudo ufw allow 80/tcp
```

HTTPS까지 쓸 예정이면:

```bash
sudo ufw allow 443/tcp
```

이렇게 외부에서 들어오는 패킷을 수신하고 HTML, CSS를 서빙할 수 있게 되었습니다.

## 3. 최소 보안 사항

### 방화벽

방화벽 설정에서 추가적으로 설정하면 좋은 부분이 있습니다. 보통 외부에서 내부로 들어오는 패킷은 차단하는 경우가 많습니다. 하지만 내부에서 외부로 들어오는 패킷은 차단하지 않는 경우가 많습니다. 하지만 보안의 관점에서 내부에서 외부로 나가는 패킷 또한 방화벽 설정을 해주는 것이 좋다고 합니다.

## 4. 나오며

지금까지 개인 서버를 구동하기 위한 인프라와 웹서버를 올리는 방법을 살펴보았습니다. IP, 포트포워딩, 라우팅 등 어려워 보이는 단어가 많습니다. 그런데 까고 보면 별 거 없습니다. 단어에 쫄지 마시고, 해당 기술이 어떤 목적으로 만들어졌는가? 어떤 기능을 하는가? 등을 살펴보시면, 추후 엔지니어링을 깊이 공부하심에 좋으리라 판단합니다. 

더하여 이후 공부해보시면 좋을 내용은 ssh 연결일 것 같습니다. 서버에서 서빙되는 정보를 수정하기 위해서 매번 서버에 방문할 필요 없이, 원격으로 서버에 접속할 수 있는 기술이 ssh입니다. 그리고 ssh 연결 시에는 꼭 fail2ban과 같은 라이브러리를 통하여 브루트포스 공격을 막으시길 바랍니다.

또한 IP로 접근하는 것이 아닌 고유의 도메인을 구매하여 사용할 수 있습니다. 이렇게 도메인에 대한 공부도 해보심이 좋을 듯 싶네요. 이때 캐싱이나 CDN에 대해 알아보시는 것도 좋습니다.

또한 도메인을 구입하셨다면 HTTPS 통신을 적용하는 것도 좋습니다. 위의 방식을 통하여 웹서버를 구동한다면 HTTP로 통신하게 됩니다. 하지만 HTTP 통신은 패킷이 암호화되지 않고 raw하게 전송된다는 문제가 있습니다. 단순히 HTML만을 서빙한다면 문제가 없어보일 수 있지만, 요새 많은 브라우저가 HTTP만을 지원하는 서버에 대하여 보안 경보를 보내어, 웹사이트에 대한 신뢰도를 떨어트릴 수 있습니다. 즉, 웹사이트의 보안과 신뢰도를 올리기 위해서는 HTTPS를 적용하는 것이 타당합니다.

먼저 웹사이트를 공인 IP로 접근해보시고 추후 쌓아가시길 바라겠습니다. 제가 존경하는 한 친구가 말하더군요. 먼저 돌아가게하라. 그리고 효율적으로 만들어라. 다음으로 유지보수 가능하게 만들어라. 아름다운 공학을 추구하시는 엔지니어가 되시길 바라겠습니다.

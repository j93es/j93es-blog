## 0. 들어가며

프론트엔드 엔지니어로써 차별점을 지닐 수 있는 방식에는 무엇이 있을까요? 화려한 UI 구현, 최신 기술 적용 등 여러 좋아보이는 방안이 있을 겁니다. 그런데 저는 조금 다르게 바라보고 싶습니다. 기본에 충실한, 목적에 부합하는 엔지니어링이 오히려 그 웹사이트를 돋보이게 만들 수 있지 않을까요? 화려함을 추구하는 오늘, 저는 정제된 아름다움을 추구하고 싶습니다.

그렇다면 정제된 아름다움이란 무엇일까요? 목적 외의 추가적인 것을 제하는 것이 중요하다고 생각합니다. 그리고 남은 리소스로 디테일을 챙기는 것이 중요한 것 같습니다. 챙겨야할 디테일을 챙기지 않고, 화려함만을 추구하는 것은 제가 추구하는 엔지니어링을 아닌 것 같습니다.

그렇다면 프론트엔드에서 챙겨야할 디테일에는 무엇이 있을까요? 제가 블로그를 개발하며 겪은 경험을 바탕으로 프론트엔드에서의 디테일에 대해 살펴보겠습니다.

## 1. 프로젝트 소개

이 글을 보고 있는 블로그가 본 포스팅에서 소개할 프로젝트입니다. 블로그의 본질은 무엇일까요? 결국 글을 읽는 것이라고 생각합니다. 그렇다면 어떤 환경이 글을 읽기에 좋은 환경일까요? 저는 책과 같은 깔끔함을 추구하는 것이 좋다고 생각합니다. 여러 블로그들을 살펴본다면, 이미지들로 인하여 조금은 난잡하다는 느낌이 드는 경우가 종종 있습니다. 그렇기에 저는 난잡함을 줄이고 깔끔하다는 인상을 주는데에 그 목적으로 맞추었습니다. 즉, 정제된, 절제된 아름다움을 추구하고자 하였습니다.

특히 완성도 있게 프론트엔드를 구성하려하였습니다. 그렇기 위해서는 프론트엔드에서 디테일을 챙기는 작업이 필요하겠지요. 이제부터는 완성도 있는 프론트엔드를 위한 디테일을 살펴보겠습니다.

## 2. favicon, logo

favicon 웹페이지에 적절히 적용하셨나요? favicon은 브라우저의 주소창이나 탭에 표시되는 이미지 입니다. 해당 이미지로 웹사이트에 대한 인상을 심어주기도 합니다. 그런데 생각보다 많은 분들께서 이러한 favicon을 기본 템플릿 그대로 사용하기도 합니다. 웹페이지가 완성도 있게 구성되었다는 느낌을 주기 위해서는 favicon 또한 잘 적용해보심이 좋을 듯 싶습니다.

logo는 사이트를 대표하는 이미지입니다. 이때 logo를 적용할 때, 고려해야하는 요소가 있습니다. 요청한 디바이스에 따라, 서빙되는 로고의 해상도를 세분화하시는 것도 좋아보입니다. 작은 기기에 큰 해상도의 logo를 전달한다면, 서빙하는 파일의 크기가 증가할 수 있습니다. 이때 logo는 html의 해더에서 파싱될때 요청됩니다. 그런데 logo의 용량이 크다면, 추후 요청될 이미지나 기타 파일의 로딩이 느려질 수 있습니다. 즉, 병목이 발생할 수 있다는 것이죠. 따라서 작은 기기에는 작은 해상도의 logo를 큰 기기에는 큰 해상도의 logo를 전송하여, 필요한만큼의 해상도를 전송할 수 있도록 하는 것이 중요할 듯 싶습니다. 다시말해 전문적으로 프론트엔드를 구성하시고 싶다면, 디바이스 별로 다른 해상도의 logo 이미지를 서빙하는 것도 좋아보입니다.

(사실 favicon을 둥글게 다듬어서 서빙하는 것이 더욱 좋다고 판단합니다. 저도 적용 안했습니다. 바빠요. ㅎㅎ)

## 3. manifest.json

프론트엔드에서의 manifest.json은 브라우저가 웹사이트를 앱처럼 설치하고 실행할 수 있도록 필요한 정보를 제공합니다. 예컨데 스마트폰에서 특정 웹페이지를 바탕화면에 추가하여 앱처럼 사용할 수 있습니다. 이때 해당 앱이 어떻게 보여질지 등을 결정하는 것이 manifest.json입니다. 더하여 logo의 종류와 해상도에 따라 어느 경로로 서빙하는지 정보를 제공하기도 합니다. 

다음은 저의 블로그에 적용된 예시입니다.

```json
{
  "short_name": "j93es blog",
  "name": "j93es blog",
  "icons": [
    {
      "src": "/logo/favicon.ico",
      "sizes": "256x256",
      "type": "image/x-icon"
    },
    {
      "src": "/logo/logo192.png",
      "type": "image/png",
      "sizes": "192x192"
    },
    {
      "src": "/logo/logo512.png",
      "type": "image/png",
      "sizes": "512x512"
    }
  ],
  "screenshots": [
    {
      "src": "/logo/logo512.png",
      "sizes": "512x512",
      "type": "image/png",
      "form_factor": "wide",
      "label": "Logo"
    },
    {
      "src": "/logo/logo512.png",
      "sizes": "512x512",
      "type": "image/png",
      "form_factor": "narrow",
      "label": "Logo"
    },
    {
      "src": "/logo/j93es-logo.svg",
      "sizes": "200x100",
      "type": "image/svg",
      "label": "Logo"
    }
  ],
  "start_url": ".",
  "display": "standalone",
  "theme_color": "#f5f8ff",
  "background_color": "#f5f8ff"
}
```

기본 템플릿 그대로 사용하시는 경우도 있으실텐데요, 한번쯤은 신경쓰셔야할 파일이라고 생각합니다. 프론트엔드에서는 단순 기능 구현 또한 중요하지만, 남들과 차별되는 강점을 가지시려면 사소한 디테일을 챙기는 것이 중요하다고 생각합니다.

## 4. noscript

여러분의 디바이스의 브라우저는 자바스크립트 실행을 허용하나요? 대부분 허용할 것입니다. 그런데 자바스크립트 실행을 허용하지 않는 경우도 있지 않을까요? 자바스크립트 실행이 허용되지 않는다면, 웹사이트가 제대로 동작하지 않을 것입니다. 그렇다면 자바스크립트 실행을 허용하지 않는 환경에서는 어떤 페이지를 전달하는 것이 타당할까요? 바로 noscript 페이지입니다. 예를 들기 위하여 index.html의 head 태그를 살펴봅시다.

```
<!DOCTYPE html>
  <head>
    <noscript>
      <meta
        http-equiv="refresh"
        content="0; url=/noscript-page/noscript.html"
      />
    </noscript>
  </head>
```

위와 같이 head 태그안의 noscript 태그가 있습니다. 만약 브라우저의 환경이 자바스크립트 실행을 허용하지 않는다면, 브라우저는 meta 태그의 경로로 noscript 페이지를 다시 요청하여 적절한 페이지를 노출시킵니다. noscript 페이지를 커스텀하여 사용하는 것 또한 고려해봄직합니다.

## 5. custom error page

에러 페이지와 기본 페이지 간의 디자인 통일성을 유지하는 것이 완성도 높은 웹페이지라고 말할 수 있지 않을까요? 간혹 발생하는 에러에 대하여 웹페이지는 에러를 알리는 페이지를 표출하기도 합니다. 그런데 리버스 프록시 유틸리티 등에서 기본적으로 제공하는 에러 페이지를 그대로 사용하는 경우 또한 있습니다. 하지만 완성도 높게 프론트엔드를 구성하시고 싶다면, 에러페이지를 커스텀하여 사용하는 것 또한 중요하다고 생각합니다.

더하여 단순히 React 내에서 에러페이지를 띄운다면, Soft 404 등의 사소한 SEO 관련 문제가 생길 수 있습니다. 즉, 에러에 대한 SEO를 고려하고 싶으시다면, 백엔드 측에서 직접적으로 에러페이지를 전달하는 것이 더욱 좋습니다. 관련하여 블로그 에러핸들링 포스팅에서 더욱 자세히 다룰 예정입니다.
